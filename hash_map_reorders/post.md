
#cpp #hashtable

Наверное многим знакома проблема с хеш-таблицами и детерменированностью их сериализации. Если менять что-нибудь в базовых либах, то быстро находятся тесты, которые дифуют из-за изменения порядка обхода элементов в хеш-таблицах. Либо при релизе чего-нибудь такого пробивает кеши, ибо система полагалась на стабильность сериализации (тут стоит вспомнить protobuf-map, где в дефолте есть рандомизация хешей - и кто-то это считает ). И там и там лезут дифы, которые по идее не должны быть значимы с точки зрения логики программы (если она правильно написана), но порядок полей едет и с точки зрения "байтовой эквивалентности" что-то всё таки изменилось.

В целом - хеш-таблицы без рандомизации нам гарантируют следующее: если засунуть`(*)` объекты в хеш-таблицу в неком порядке два раза, то перебор из (begin->end) будет совпадать. У меня возник вопрос - а гарантируется ли ещё что-нибудь? Например - как соотносятся порядок элементов begin->end если во второй раз класть их в том порядке, который образовался после первого раза.

Как соотносятся порядки элементов между `x` на входе функции и `res` на выходе `RefillSimple`?
```
template<class T>
T RefillSimple(const T& x) {
    T res;
    for(auto& p : x) {
        res[p.first] = p.second;
    }
    return res;
}
```

Здесь и далее я проверяю на 100к случайных элементах (с фиксированным сидом, вышло 99'996 уникальных значений - в соответствии с парадоксом дней рождений).
```
void InitialFill(auto& x) {
    srandom(Seed);
    for(size_t i = 0; i < ElemsToStore; ++i) {
        x[random()] = i;
    }
}
```

Измерять будем количество нарушений монотонности (в сравнении с исходным порядком в хеш-таблице после завершения добавлений). Количество нарушений в самой исходной хеш-таблице получилось 49'568 - просто видим что почти с 1/2 вероятностью следующее число будет больше или меньше предыдущего. Так вот - на выходе RefillSimple вышло 43'887 нарушений. Стат-значимо отличается, но в практическом смысле "нет никакой закономерности".

Откуда вообще предположение что должна быть какая-то связь и что-то должно сохраняться? А идея то очень простая: у нас ведь у данных не меняется ключ, то есть в бакет данные попадут тот же бакет, то есть состав бакетов должен воспроизвестись. Замечу, что число бакетов на входе и выходе у RefillSimple действительно одинаковое (ибо мы проходили такой же набор проверок на сравнении с load_factor и в тех же точках делали увеличение размера с такой же политикой его изменения).

Интересное открытие - если добавить `res.reserve(x.size())`, то число нарушений монотонности станет вообще 74'304. Но тут загадки нет - в `x` у нас число бакетов 172'933, а в res если делать reserve - только 107'897 (в std на котором я тестирую (то что само пришло с клангом) - max_load_factor в дефолте равен 1).

Ок, давайте починим последнее - делаем reserve с запасом, указывая вместо числа элементов - число бакетов `res.reserve(x.bucket_count() * x.max_load_factor)` (или просто rehash) и ... получаем `99'995` нарушений - на 1 меньше числа элементов - теоретический максимум. Другими словами - мы получили просто обратный порядок элементов.

- `Утверждение 1` - если переложить данные из одного unordered_map в другой, заранее выровнив число корзин, то порядок получится обратным.
- `Следствие` - если сделать дважды такое перекладывание - то порядок совпадёт (`unordered_map - reservedRefillRefill MissOrders 0` - в [report.txt](https://github.com/ilnurkh/blog/blob/29cbb6be01f1df87179dd3325952bcc83383b2fa/hash_map_reorders/report.txt#L14))

Итак, мы имперически узнали, что 1) данная реализация unordered_map внутри бакетов держит односвязный список элементов, добавляя новые в начало (стек внутри бакетов в общем) 2) сами бакеты провязываются в список также в порядке обратном их первому использыванию. Из этого можно наверное уже предположить почему у нас были отклонения от случайности в первой версии переноса (43к) и во второй (74к). Если сделать rehash с повышением - то порядок почти меняется на обратный (примерно 80% отклонений), в ходе набора целевого состояния такой рехеш какое-то число раз случался.

Можно глянуть и на какие-то другие хеш-мапы. Например аркадийный [THashMap](https://github.com/catboost/catboost/blob/c828709e1e37c95a5cf168d81248096aa77e381d/util/generic/hash.h#L9) бакеты провязывает в прямой список и там поведение совсем [другое](https://github.com/ilnurkh/blog/blob/main/hash_map_reorders/report_arcadia.txt).


`(*)` Одинаковым кодом, как ясно из текста - всякие вызовы reserve тут существенны.

п.с. захотелось добавить пару слов о таблицах с открытой адресацией - тут всё запутаннее, и исходному тезису (о фиксированности порядка) в большинстве реализаций (обычно в флэт-хеш-таблицах, где мы бежим по исходному массиву (не поддерживается список заполненных ячеек) таблице и смотрим - заполнена ли ячейка или нет) неоткуда взяться вовсе - слишком сильно эволюция состояния влияет на конкретное расположение элементов и на порядок их перебора.
